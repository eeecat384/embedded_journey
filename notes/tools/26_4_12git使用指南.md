# git多人合作管理流程

## 0.git版本管理文件结构
disk：本地磁盘存储的文件，内容为git当前所处分支的文件<br>
local(git端)：本地的git仓库，有分支，历史版本等<br>
remote(github端)：远程仓库，为远端的git<br>

在此例子中，remote有一个main（更改部分将以代码块的形式标出）

>**Remote** (origin)
>- main(master): Init
>
>**Local** (git)
>
>（空）
>
>**Disk** (Working Directory)
>
>（空）

## 1. 克隆到本地
GitHub Desktop（下称本地）顶部左边的Current repository->add->clone reposity

或使用命令`git clone https://github.com/你的用户名/项目名.git`

>**Remote** (origin)
>- main(master): Init
>
>**Local** (git)
>- `main(master)`: `Init`
>
>**Disk** (Working Directory)
>- `main(master)` `Init`

## 2. 建立新的feature branch
本地顶部中间Current branch->New branch添加新的my-branch将main复制为一份feature

或使用命令`git checkout -b my-feature`

>**Remote** (origin)
>- main(master): Init
>
>**Local** (git)
>- main(master): Init
>- `my-feature`: Init 
>
>**Disk** (Working Directory)
>- `my-feature`: Init

## 3. 修改，add，commit并push文件
修改后在本地添加描述并点击“Commit x file to **my-featrue** ”

或使用命令
```bash
git diff                 #建议先查看有哪些变化
git add <change_file>    #这里的是所有想要修改的文件的文件名
git commit -m "feat: 添加新功能描述"
```
许多项目都使用Conventional Commits来规范提交信息，基本格式如下，详见附录
```plain
<type>(<scope>): <简短描述>
[详细说明（可选）]
[脚注/BREAKING CHANGE（可选）]
```

在本地点击push按钮或者使用快捷键ctrl+P以push文件

或使用命令`git push origin my-featrue`

>**Remote** (origin)
>- main(master): Init -> `update`
>- `my-feature`: Init -> `f-commit`
> 
>**Local** (git)
>- main(master): Init
>- my-feature: Init -> `f-commit`
>
>**Disk** (Working Directory)
>- my-feature: Init -> `f-commit`

## 4. 将远端更新同步至本地
**在把my-feature push到github后，通常会出现main branch又有更新的情况，所以我们首先需要切换到main：**

在本地顶部中间Current branch->main 或使用命令`git check out main`

**然后将远端的main拉取到local的main中：**

在本地顶部右边点击“pull origin”（如无更新则会显示“fetch origin ”） 或使用命令`git pull origin main`

>**Remote** (origin)
>- main(master): Init -> update
>- my-feature: Init -> f-commit
> 
>**Local** (git)
>- main(master): Init -> `update`
>- my-feature: Init -> f-commit
>
>**Disk** (Working Directory)
>- `main(master)`: `Init` -> `update`

## 5. 同步main和feature代码变化
**首先将本地分支切换到my-feature**

在本地顶部中间Current branch -> my-feature

或者使用命令`git checkout my-feature`

**当前状态的源代码就是有自己修改的f-commit的变化，但是没有远端update的变化**

**为了同步这俩的代码改变,进行如下操作**

点击顶部菜单栏的Branch -> Rebase current branch<br>
在弹出的对话框中选择 main 作为目标分支，点击Start rebase

或使用命令git rebase main

**这里的操作就是把update后的main复制过来，尝试将f-commit弄到最新的main上**

>**Remote** (origin)
>- main(master): Init -> update
>- my-feature: Init -> f-commit
> 
>**Local** (git)
>- main(master): Init -> update
>- my-feature: Init -> `update` -> `f-commit`
>
>**Disk** (Working Directory)
>- `my-feature`: `Init` -> `update` -> `f-commit`

## 6. push到github并提交拉取申请(Pull Request)

**执行完 rebase 后，GitHub Desktop 会自动检测到需要强制推送，push按钮会变色**

点击Force push origin按钮

或使用命令`git push -f origin my-feature`

push成功后，在GitHub页面操作：
1. 访问你的Fork仓库 → 点击 **"Compare & pull request"**
2. **base repository**: 原项目 `原作者/项目名` 的 `main` 分支
3. **head repository**: 你的Fork `你的用户名/项目名` 的 `feature/分支名`
4. 填写清晰的PR描述：
   - **标题**：简洁说明修改内容
   - **描述**：详细说明修改原因、测试情况、关联的Issue编号

>- main(master): Init -> update
>- my-feature: Init -> `update` -> `f-commit` ——>PR
> 
>**Local** (git)
>- main(master): Init -> update
>- my-feature: Init -> `update` -> `f-commit`
>
>**Disk** (Working Directory)
>- `my-feature`: `Init` -> `update` -> `f-commit`

维护者会审查代码，可能提出修改意见，你可以继续向同一分支推送提交，PR会自动更新

审查通过后，维护者会选择合并方式：

**Squash and merge**：压缩为一个干净提交

**Create merge commit**：保留完整历史

**Rebase and merge**：变基合并保持线性历史

通常会选择Squash and merge来让同一时间的PR作为一个commit以保持提交记录整洁

PR被合并后，清理本地分支和远程my-feature分支


## 7. 完整流程

| 步骤       | 命令                                       | 说明                 |
| ---------- | ------------------------------------------ | -------------------- |
| **克隆**   | `git clone https://github.com/你/项目.git` | 下载到本地           |
| **切分支** | `git checkout -b my-feature`               | 创建功能分支         |
| **开发**   | 编辑代码                                   | 在功能分支上工作     |
| **提交**   | `git add . && git commit -m "feat: xxx"`   | 规范提交信息         |
| **推送**   | `git push origin feature/xxx`              | 推送到你的Fork       |
| **PR**     | 网页操作                                   | 向原项目发起合并请求 |
| **审查**   | 与维护者对话修改                           | 根据反馈调整代码     |
| **合并**   | 维护者操作                                 | 维护者合并后清理分支 |

## 附录1：Conventional Commits使用指南

规范结构
```plain
<type>(<scope>): <简短描述>
[详细说明（可选）]
[脚注/BREAKING CHANGE（可选）]
```

**1. Type（必须）**
提交的类型，常见类型包括：

| 类型       | 含义                                               |
| ---------- | -------------------------------------------------- |
| `feat`     | 新功能（feature）                                  |
| `fix`      | 修复 bug                                           |
| `docs`     | 文档变更                                           |
| `style`    | 代码风格调整（格式化、缺失分号等，不影响代码逻辑） |
| `refactor` | 代码重构（既不修复 bug 也不添加功能）              |
| `perf`     | 性能优化                                           |
| `test`     | 测试相关（添加缺失的测试、修正现有测试）           |
| `build`    | 构建系统或外部依赖的变更                           |
| `ci`       | CI 配置变更（GitHub Actions、Travis、Circle 等）   |
| `chore`    | 其他不修改 src 或测试文件的变更                    |
| `revert`   | 回滚之前的提交                                     |

**2. Scope（可选）**

表示提交影响的范围，如模块名、文件名或功能区域：
```
feat(parser): add ability to parse arrays
```

**3. Description（必须）**

提交的简短描述，使用现在时态（如 "change" 而非 "changed"），首字母小写，结尾不加句号。

**4. Body（可选）**

详细描述变更的动机和对比之前行为的差异，可以分多行。

**5. Footer（可选）**

用于引用 Issue 或标记破坏性变更（Breaking Changes）。


## 附录2：实际开发项目可能使用到的分支：（来自网络）

main：生产环境，也就是你们在网上可以下载到的版本，是经过了很多轮测试得到的稳定版本。

release： 开发内部发版，也就是测试环境。

dev：所有的feature都要从dev上checkout。

feature：每个需求新创建的分支。


下面介绍一下一个新需求过来的git操作流程：

1.从dev分支上checkout -b new-feature，进行开发

2.开发完后，经过自测没问题了，准备发版

3.merge到release分支上进行发版，并打tag

4.有bug就直接在release上进行修改，改完再次发版，打tag，直到测试人员验证完毕

5.这时可以将release分支合并到dev上，也可以删除掉feature分支了，并等待通知是否将此功能上线（发布正式版本，也就是在正式服务器上运行），如果上线，那就merge到main（master）分支，当然了有可能需要将ip改为生产服务器的地址，并打上一个official tag。

6.如果上线后，发现有bug，如果此时main分支已经又合并上新功能B了，但是这个临时的版本又不想包含B的功能，那么可以切换到上次的official tag，checkout -b一个hotfix分支进行bug修复，hotfix分支要进行保留，不能删除。测试没问题就可以发版了，最后可以合并到main上。

一般的项目这套流程应该就足够了。所有的发版tag都会集中到release，main，hotfix分支上，所有的需求都会从dev上拉取，这样能保证代码是完整可用的，是经过每次release合并过来的
